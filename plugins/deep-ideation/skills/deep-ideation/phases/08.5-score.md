# Phase 8.5: SCORE

Launch the Scorer. See `agents/scorer.md`.

## When to Run

After Phase 8 (SYNTHESIZE) and before Phase 9.5 (STRESS-TEST). Runs in ALL modes.

The Synthesizer defines criteria and weights. The Scorer applies them. This separation prevents the idea-generating agent from scoring its own output.

## What It Receives

- `$WORKSPACE/08-synthesize.md` — the Synthesizer's criteria list, weights, and rationale
- `$WORKSPACE/ideas.csv` — every idea from the session
- `$WORKSPACE/01-discover.md` — root causes (for context on what criteria mean)
- `$WORKSPACE/07-tension.md` (if it exists) — session tensions

## What It Produces

1. **Updated `ideas.csv`** — new/filled columns:
   - criteria columns (filled by the Scorer, e.g., `feasibility`, `novelty`, and session-specific criteria)
   - `total_score` (weighted average using Synthesizer's weights)
   - `menu_bucket` (qualitative label: `quick_win`, `core_bet`, `moonshot`, or empty)

2. **Ranked output** saved to `$WORKSPACE/08.5-score.md`:
   - Full ranked table
   - Idea Menu with qualitative bucket assignments
   - Notes on unscored ideas, if any

## What It Does NOT Do

- Does NOT generate new ideas
- Does NOT redefine criteria or weights (those belong to the Synthesizer)
- Does NOT invent business precision (market size, effort days, revenue)

## Key Commands

```bash
# Discover current schema — confirm criteria columns exist from Synthesizer
python scripts/idea_db.py describe <workspace>

# If criteria columns are missing (shouldn't be), add them
python scripts/idea_db.py add_criteria <workspace> \
  --criteria "feasibility,novelty,[session-criteria]" \
  --composite "total_score"

# Score every idea (Scorer produces scores.json)
python scripts/idea_db.py set_batch <workspace> scores.json

# Compute the weighted total
python scripts/idea_db.py compute <workspace> \
  --criteria "feasibility,novelty,[session-criteria]" \
  --target "total_score" \
  --formula weighted_avg \
  --weights "feasibility:2,novelty:3,[root_cause_crit]:4,[tension_crit]:2"

# Apply mechanism_density penalty: multiply total_score by 0.5 for low-density ideas
python scripts/idea_db.py apply_penalty <workspace> \
  --filter "mechanism_density=low" \
  --target "total_score" \
  --multiplier 0.5

# Add menu_bucket and assign buckets to top candidates
python scripts/idea_db.py add_column <workspace> menu_bucket --default ""
python scripts/idea_db.py set <workspace> <id> menu_bucket "quick_win"
python scripts/idea_db.py set <workspace> <id> menu_bucket "core_bet"
python scripts/idea_db.py set <workspace> <id> menu_bucket "moonshot"

# Export ranked table for the output file (include mechanism_density column)
python scripts/idea_db.py export_md <workspace> \
  --columns "id,name,source_agent,temperature_zone,mechanism_density,total_score,menu_bucket" \
  --sort total_score --desc

# Log telemetry
python scripts/idea_db.py telemetry <workspace> baseline_count \
  $(python scripts/idea_db.py count <workspace> --filter "tag=BASELINE")
```

## Expected Output

Save to `$WORKSPACE/08.5-score.md`. This is the authoritative ranking for downstream phases.

## Transition

After Scorer completes:
- **STANDARD/DEEP:** proceed to Phase 9.5 STRESS-TEST (which reads `top <workspace> total_score --n 5`)
- **LITE:** proceed directly to Phase 10 BRILLIANCE FILTER

## Anti-Patterns

- **Don't let the Scorer redefine criteria.** The Synthesizer owns criteria selection; the Scorer owns application. Mixing the two reintroduces the bias the split was designed to remove.
- **Don't bucket every idea.** Most ideas should have `menu_bucket` empty. A bucket is a recommendation, not a label. Aim for 3-5 per bucket, maximum.
- **Don't use the ICE formula.** ICE is gone — `total_score` is a weighted average of session-derived criteria, not `(Impact × Confidence) / (11 - Ease)`.
- **Don't skip the mechanism_density penalty.** Ideas tagged `mechanism_density=low` are slogan-level, not implementation-level. Without the 0.5× penalty they compete unfairly against ideas with real mechanisms, turning the final ranking into a cleverness contest.
- **Don't skip baseline presence check.** If no `[BASELINE]` ideas exist in the DB, flag this to the operator before proceeding — the scoring has no reality anchor.
